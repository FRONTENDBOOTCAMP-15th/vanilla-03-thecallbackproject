# [비동기들 : The Callback Project](https://thecallbackproject.netlify.app/)

<br><br>

## 📑 프로젝트 개요

기본 웹 기능을 제대로 익히고 구현하는 데 집중하기 위해 ‘브런치’ 시안을 선택했습니다.  
필수 기능을 먼저 안정적으로 완성하고, 선택 기능들도 하나씩 시도해 보면서 자연스럽게 확장되는 구조를 목표로 하였습니다.
<br>
<br>

## 🎬 프로젝트 기간

> 2025.11.10 ~ 2025.11.24

## ⚙️ 트러블 슈팅

<table>
  <tr>
    <th>이름</th>
    <th>문제점</th>
    <th>해결 사례</th>
  </tr>

  <!-- 소혜림 -->
  <tr>
    <td><b>혜림</b></td>
    <td>
      <b>1.</b> 파일 서버 타임아웃으로 이미지 업로드 실패<br>
      <b>2.</b> 렌더링 과정에서 ARIA 접근성 속성 누락<br>
      <b>3.</b> 이미지 첨부 취소 기능을 뒤늦게 인지<br>
      <b>4.</b> 태그 입력창 자동완성 비활성화 누락<br>
      <b>5.</b> 최근 본 글 저장 로직 뒤늦게 인지<br>
      <b>6.</b> 모바일 가상 키보드 노출 시 툴바가 가려지는 이슈
    </td>
    <td>
      <b>1.</b> 업로드 시간을 줄이기 위해 2MB 제한을 임시 적용<br>
      <b>2.</b> innerHTML/createElement로 생성 시 ARIA 속성을 함께 추가하도록 로직 수정<br>
      <b>3.</b> input 초기화 + 미리보기 제거 기능 구현<br>
      <b>4.</b> autocomplete="off" 추가로 자동완성 비활성화<br>
      <b>5.</b> 상세페이지 진입 시 ID·제목 등 필요한 데이터 로컬스토리지 저장 기능 구현<br>
      <b>6.</b> visualViewport height 변화 감지해 툴바 위치를 동적으로 조정하여 해결
    </td>
  </tr>

  <!-- 오지아 -->
<tr>
    <td><b>지아</b></td>
    <td>
      <b>1.</b> 로그인 상태인데도 로그아웃 UI만 출력됨<br>
      <b>2.</b> 프로필 드롭다운 기능의 UI 제어 구조가 불안정함<br>
      <b>3.</b> API 요청 시 데이터 불러오기 실패<br>
      <b>4.</b> 빌드 환경에서의 이미지 불러오기 깨짐
    </td>
    <td>
      <b>1.</b> 로컬스토리지 사용자 정보 구조 통일 및 accessToken 기반 UI 분기 수정<br>
      <b>2.</b> 단일 return → if 조건 구조로 변경해 상태별 UI 안정적 제어<br>
      <b>3.</b> 일치하지 않는 요청 파라미터 이름 통일하여 문제 해결<br>
      <b>4.</b> onerror/직접 경로 입력을 import 방식으로 변경해 빌드 시 자동 경로 변환
    </td>
  </tr>

  <!-- 박예슬 -->

    <td><b>예슬</b></td>

      <b>3.</b> API·Axios 이해 부족으로 개발 흐름 파악 어려움<br>

    <td>

## 🟣 회고 : 배운 점, 아쉬운 점, 만족한 점

### 🪻 소혜림

이번 프로젝트를 통해 팀 작업은 개인 의욕만으로 밀어붙인다고 굴러가는 것이 아니라,<br>각자의 상황과 속도를 이해하며 함께 맞추는 과정이 중요하다는 것을 배웠다.
<br>초기 설계 단계에서 모듈화를 충분히 하지 않아 뒤늦게 더 큰 수정 비용을 치른 경험은 “구조는 처음부터 잡아야 한다”는 교훈을 남겼다.
<br>한편 배포를 제때 진행하지 못해 일정이 촉박해졌고, 그로 인해 추가 기능을 시도하지 못한 점은 아쉬움으로 남지만,
<br>API를 받아 화면에 렌더링하기까지의 전체 흐름을 직접 구현하며 데이터 기반 UI의 구조를 온전히 이해하게 된 점은 큰 수확이었다.
<br>

### 🪻 오지아

막연히 이해했다고 생각했던 개념들도 실제 코드에 적용하는 과정에서 어려움을 겪으며, 직접 구현을 통해 이해가 깊어진다는 사실을 다시 느꼈다.
<br>마크업부터 API 렌더링까지 전반적인 흐름을 따라가며 앞으로 어떤 방향으로 학습해야 할지 스스로 점검할 수 있었다.
<br>또 배포 과정에서 팀원들의 조언과 해결 방식을 가까이서 지켜보며 팀 프로젝트 속에서만 얻을 수 있는 실제 흐름을 확인할 수 있었다.
<br>길지 않은 기간이었지만 내가 놓치고 있던 부분을 돌아보게 해준 의미있는 시간이었다.
<br>이번에 부족하다고 느낀 부분들은 보완해 다음 프로젝트에서는 더 나은 결과를 내고 싶다.
<br>

### 🪻 박한섭

프로젝트를 진행하면서 전반적으로 내가 부족한 부분이 많다는 걸 다시 느꼈고, 팀원들과의 소통 방식부터 코드 구조 잡는 방법까지 여러 면에서 큰 자극을 받았다.
<br>특히 모듈화와 UI 컴포넌트화를 제대로 적용하지 못한 아쉬움이 컸고, 그 중요성을 직접 체감할 수 있었다. 비동기 처리나 로컬스토리지 같은 기능들도 구현하면서 점차 자연스러워졌지만, API와 비동기 로직은 더 많이 다루고 싶다는 욕심이 생겼다.
<br>아직 구현 능력과 이해도가 부족해 미완성된 기능들이 남았지만, 이번 경험 덕분에 앞으로 어떤 방향으로 학습해야 하는지 더 분명하게 알게 되었다.
<br>완벽하진 않았지만 여러 기능을 직접 구현하며 이해가 깊어진 부분들도 있었고, 그 과정이 즐거웠다는 점에서 의미 있는 시간이었다.

### 🪻 박예슬

이번 프로젝트를 진행하면서 이건 이해했지 하고 넘겼던 부분들에 어려움을 겪는 내 자신을 보고 개념만 배우는 것과 직접 구현 해본다는 게 얼마나 큰 차이인지 알게되었다.
<br>마크업 부분에서도 분명 더 좋은 코드들이 있었는데 기억을 해내지 못하고, 더 꼼꼼하게 봤어야 하는 부분들을 챙기지 못 한 것에 대해 매우 아쉬운 것 같다.
<br>아직 이해도와 능력치가 부족해서 시안과 똑같이 구현하진 못했지만 이번 프로젝트를 통해 어느 부분이 부족했고, 어느 부분을 더 공부해야 하는 지 확실하게 깨달은 것 같다.
<br>다음 프로젝트 땐 같은 실수를 반복하지 않기 위해 이번 경험을 머릿속에 넣고 잊지 않을 것이다.
